# Clean Architecture & Clean Code 가이드

## 1. 우리 프로젝트에서의 클린 아키텍처

토이 프로젝트라 엔터프라이즈급 4-레이어를 강요하지 않는다. 대신 **핵심 원칙**만 지킨다.

### 의존성 방향 (한 줄로)
```
UI(Svelte 컴포넌트) → Application(유스케이스/서비스) → Domain(엔티티/규칙) ← Infrastructure(데이터/API)
```
- **안쪽(Domain)은 바깥(UI/Infra)을 모른다.**
- 비즈니스 규칙은 프레임워크·DOM·HTTP·DB를 직접 참조하지 않는다.

### 디렉토리 가이드 (예시)
```
src/
├── lib/
│   ├── domain/          # 순수 타입·규칙 (Restaurant, SwipeAction 등). Svelte/DOM 의존 금지.
│   ├── application/     # 유스케이스 (스와이프 처리, 킵 추가 등). Domain만 의존.
│   ├── infrastructure/  # 데이터 fetch, 로컬 저장소, API. 외부 세계와의 경계.
│   └── ui/              # Svelte 컴포넌트. 위 레이어를 호출만 한다.
└── routes/              # SvelteKit 라우팅 (선택)
```

> 토이 프로젝트 단계에서 디렉토리를 다 만들지 않아도 된다. **레이어를 의식하며 분리**하는 것이 핵심.

## 2. 클린 코드 원칙 (이 프로젝트에서 반드시 지킬 것)

### 네이밍
- **의도가 드러나는 이름**: `d`, `temp`, `data2` 금지. `daysUntilExpiry`, `currentRestaurant`처럼.
- **부울은 `is/has/can`** 접두어: `isLoading`, `hasNextCard`.
- **함수는 동사구**: `swipeRight()`, `loadRestaurants()`.

### 함수
- **하나의 책임**: 함수가 "그리고"로 설명되면 쪼갠다.
- **짧게**: 화면 한 눈에 들어오는 길이를 목표.
- **부수효과 최소화**: 가능한 한 순수함수로. 부수효과가 있으면 이름에서 드러나게.

### 주석
- 기본 정책: **주석은 쓰지 않는다.**
- 예외: WHY가 비명백할 때만. (예: 특정 브라우저 버그 회피, 외부 제약)
- WHAT을 설명하는 주석 금지 (이름이 못한 일을 주석이 대신할 수 없다).

### 매직 넘버 / 문자열
- 의미 있는 상수로 추출: `const SWIPE_THRESHOLD_PX = 100;`

### 에러 핸들링
- 시스템 경계(네트워크/유저 입력/외부 API)에서만 방어. 내부 호출은 신뢰한다.
- 무의미한 try-catch 금지.

## 3. Svelte 특화 가이드

- **컴포넌트는 표현(Presentation)에 집중**한다. 비즈니스 로직은 `lib/application`으로 빼낸다.
- **Stores**는 Application 레이어의 상태를 노출하는 통로로 사용. 컴포넌트가 직접 fetch/저장하지 않는다.
- **Props 인터페이스를 명시**한다 (TypeScript 권장 — ADR로 결정).

## 4. 토이 프로젝트 현실 노트

- 이 가이드는 **방향**이지 **체크리스트**가 아니다.
- 한 화면짜리 MVP에서 4레이어를 다 만들 필요는 없다. 하지만 **"나중에 분리할 수 있게"** 짠다.
- 의심 시: "이 코드가 6개월 뒤 내가 봐도 이해되는가?" — 답이 NO면 리팩토링.
